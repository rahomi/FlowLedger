# Architecture Overview

This document describes the high‑level architecture of the **Offline‑First Personal & Small‑Business Finance Manager**.

---

## 1. Monorepo Layout

We use a **monorepo** to keep the backend, frontend, shared libraries, and documentation in a single repository. The top‑level structure mirrors the recommended layout for NestJS + Vite + TypeScript projects.

```
.
├── apps/                     # Deployable applications
│   ├── api/                  # NestJS backend (REST API)
│   │   ├── src/              # Application source code
│   │   ├── test/             # Integration / unit tests
│   │   └── tsconfig.json
│   └── web/                  # React + Vite frontend
│       ├── src/              # React entry point and UI components
│       ├── public/           # Static assets (icons, manifest, etc.)
│       └── vite.config.ts
│
├── packages/                # Shared code and utilities
│   ├── db/                   # TypeORM entities & migrations (shared with API)
│   ├── dto/                  # TypeScript interfaces shared between client & server
│   ├── types/                # Common types (e.g., enums, constants)
│   └── utils/                # Helper functions (date utils, formatting, etc.)
│
├── docs/                     # Architecture, design, and process docs (this folder)
│
├── scripts/                  # Build / CI helper scripts
│   └── sync-indexeddb.ts     # Offline sync bridge (web ↔ IndexedDB ↔ API)
│
├── .gitignore
├── tsconfig.json             # Base TS config for the whole repo
├── package.json              # Workspace scripts and dev dependencies
└── README.md
```

**Why a monorepo?**
- Guarantees a single source of truth for data models (`packages/db` & `packages/dto`).
- Simplifies versioning of shared types across client & server.
- Enables consistent linting, formatting, and testing across the stack.
- Facilitates CI pipelines that can build, test, and lint the whole product in one job.

## 2. Backend Architecture (NestJS)

- **Modules**: `Businesses`, `Profiles`, `Transactions`, `Loans`, `Attachments`, `Reports`, `Dashboard`, `Auth` (future).
- **Core concepts**:
  - **Controllers** expose REST endpoints (versioned under `/api/v1`).
  - **Services** contain business logic, validation, and orchestration.
  - **Repositories** (TypeORM) provide access to PostgreSQL tables.
  - **Global pipes / guards** enforce validation, transformation, and (future) authentication.
  - **Event Bus** (NestJS `EventEmitter`) for async notifications – e.g., after a transaction is created we emit `transaction.created` to update cached dashboard aggregates.
- **Database**: PostgreSQL with a schema defined in `packages/db`. All entities are versioned with TypeORM migrations.
- **Offline‑first support**: The API is stateless; the client can operate on a locally‑mirrored DB and later sync via a bulk‑upsert endpoint (`POST /api/v1/sync`).

---

## 3. Frontend Architecture (React + Vite)

- **State Management**: React Query for data fetching & caching; Dexie.js for IndexedDB persistence.
- **UI Library**: Tailwind CSS with a custom design‑system built on top of utility‑first classes.
- **Charts**: Chart.js wrapped in reusable `<Chart>` components.
- **Routing**: React Router v6 with code‑splitting for each main screen.
- **Offline Layer**:
  - All CRUD operations are first performed against Dexie.
  - A background sync worker (`scripts/sync-indexeddb.ts`) monitors network status and pushes local changes to the API.
  - Conflict resolution follows **last‑write‑wins** with timestamps; UI shows a warning for overdue conflicts.
- **Progressive Web App (PWA)**: Service worker generated by Vite‑PWA plugin, providing installable offline experience.

## 5. Offline‑First Data Flow

1. **Create / edit** a transaction → stored in IndexedDB (Dexie) and marks record as `pendingSync`.
2. **Sync worker** detects online status → batches all pending records and calls `POST /api/v1/sync`.
3. Server processes the batch within a DB transaction, returning the final persisted IDs and timestamps.
4. Client updates local records with server‑generated IDs and clears `pendingSync` flag.
5. UI reflects sync status via a small indicator in the top bar (`[Sync: ✅|⏳|⚠️]`).

The sync design is deliberately **idempotent** – the same payload can be retried safely if the server response is lost.

## 7. Non‑Functional Concerns

- **Performance**: Backend reads use indexes defined in `DATABASE.md`. Frontend employs virtual scrolling for lists >50 items.
- **Security**: All API traffic is HTTPS‑only (NFR‑4.1). Input validation is enforced via class‑validator pipes. File uploads are scanned server‑side for MIME‑type and size limits.
- **Scalability**: The PostgreSQL schema uses UUID primary keys and includes indexes for the most filtered columns (date, type, business, profile). Horizontal scaling is supported via stateless API nodes behind a load balancer.
- **Maintainability**: Modular NestJS design, exhaustive type‑safe DTOs, and automated testing (unit >80%, integration >60%).
